PIPELINE: DEDUPLICATION AND KNOWING PROCESSED RECORDS (WITHOUT data.json)

1. DEDUPLICATION STRATEGY (POWER / NETWORK FAILURES)

All ingest steps use UPSERT (INSERT ... ON CONFLICT ... DO UPDATE), so re-running the same or overlapping data is safe and idempotent.

  - Encounters:  ON CONFLICT (lab_no) DO UPDATE
  - Patients:    ON CONFLICT (lab_number) DO UPDATE
  - Test records: ON CONFLICT (encounter_id, test_name) DO UPDATE

If the pipeline fails mid-run (power, network, crash):
  - Next run fetches again from LIMS (see "Processed records" below).
  - Transform runs on the newly fetched chunk.
  - Ingest runs: existing rows are updated, new rows are inserted. No duplicate rows are created.

There is no separate "deduplication" step; the natural keys (lab_no, lab_number, encounter_id+test_name) and upserts provide deduplication.

2. LOCK TO AVOID CONCURRENT RUNS

  - The Python fetcher uses a lock file: backend/.lims_fetch.lock
  - If a run crashes without releasing the lock, the next run treats the lock as stale after 2 hours and removes it.
  - This avoids two pipeline runs writing at the same time.

3. KNOWING PROCESSED RECORDS (WITHOUT data.json)

We do not persist data.json. "Processed" state is inferred from:

  A) Fetch start date (what to ask LIMS for)
     - Stored in: backend/.last_run (timestamp of last successful fetch run)
     - Optional: backend/.last_comprehensive_run (date of last full-day sweep)
     - The Python fetcher uses .last_run to decide the start_date for the next incremental fetch. It does not need data.json for this.

  B) What is already in the database
     - The database is the source of truth for "what we have processed."
     - Encounters, patients, and test_records are upserted by natural key; re-ingesting the same record only updates it.

  C) Optional future optimization: high-water mark from DB
     - To avoid re-fetching old data after a long outage, the pipeline could query the DB for max(encounter_date) or max(time_in) and pass that as the fetch start date. Today we rely on .last_run written by the fetcher after each successful run.

4. FILES WE DO NOT TRACK (in .gitignore)

  - backend/.last_run
  - backend/.last_comprehensive_run
  - backend/.lims_fetch.lock
  - frontend/public/last_run.txt (timeout script)
  - frontend/public/processed_invoice_numbers.json (legacy; in-memory pipeline does not use it)
  - frontend/public/data.json, patients_dataset.json, tests_dataset.json

These are runtime state only; the canonical state is the database.
